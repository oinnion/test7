# Контрольні питання

### **1. Яка різниця між Git та GitHub?**

-   **Git** --- локальна система контролю версій.\
-   **GitHub** --- хостинг Git-репозиторіїв та платформа для співпраці.

### **2. Що означає "розподілена" система контролю версій?**

Кожен розробник має повну копію репозиторію й може працювати без
центрального сервера.

### **3. Яка різниця між `git init` та `git clone`?**

-   `git init` --- створює новий порожній репозиторій.\
-   `git clone` --- копіює вже існуючий репозиторій.

## 2. Staging Area та Commits

### **4. Що таке Staging Area і навіщо вона потрібна?**

Зона, де зміни готуються перед комітом; дозволяє вибрати, що увійде до
коміту.

### **5. Яка різниця між `git add .` та `git add -A`?**

-   `git add .` --- додає нові та змінені файли (не завжди видалені).\
-   `git add -A` --- додає всі зміни, включно з видаленнями.

### **6. Як написати гарне commit message?**

-   Короткий заголовок (до 50 символів).\
-   Опис *чому*, а не лише *що*.\
-   Імперативний стиль (Add, Fix, Update,...).

### **7. Як виправити останній коміт?**

    git commit --amend

## 3. Гілки та злиття

### **8. Що таке гілка в Git і чому вони "легковажні"?**

Гілка --- це вказівник на коміт; легковажні, бо займають мінімум місця й
створюються миттєво.

### **9. Яка різниця між `git merge` та `git rebase`?**

-   `merge` --- створює merge-коміт.\
-   `rebase` --- переписує історію, робить її лінійною.

### **10. Що таке fast-forward merge?**

Злиття, коли цільова гілка просто пересувається вперед без merge-коміту.

### **11. Коли виникають merge конфлікти і як їх уникнути?**

Конфлікти виникають, коли змінено ті самі рядки файлів у різних гілках.\
Уникнення: часті pull, маленькі гілки, синхронізація команди.

### **12. Чому не можна робити rebase публічних комітів?**

Бо він змінює історію, що спричиняє проблеми для інших розробників.

## 4. Віддалені репозиторії

### **13. Що таке `origin`?**

Стандартна назва віддаленого репозиторію.

### **14. Яка різниця між `git fetch` та `git pull`?**

-   `fetch` --- лише завантажує зміни.\
-   `pull` --- завантажує і одразу зливає.

### **15. Що робить `git push --force` і чому це небезпечно?**

Перезаписує історію у віддаленому репозиторії. Небезпечно, бо можна
стерти чужі коміти.

### **16. Як видалити віддалену гілку?**

    git push origin --delete branch-name

## 5. Скасування змін

### **17. Яка різниця між `git reset --soft`, `--mixed` та `--hard`?**

-   **soft** --- відміняє коміт, залишає зміни у staging.\
-   **mixed** --- відміняє коміт, зміни лишаються в working directory.\
-   **hard** --- повністю скасовує всі зміни.

### **18. Яка різниця між `git reset` та `git revert`?**

-   `reset` --- переписує історію (небезпечно).\
-   `revert` --- створює новий коміт, який відміняє зміни (безпечний).

### **19. Як скасувати `git add`?**

    git reset filename

### **20. Як повернути видалений файл?**

    git checkout HEAD -- filename

## 6. Безпека

### **21. Що не повинно потрапляти в Git репозиторій?**

-   Паролі\
-   Токени\
-   Ключі\
-   `.env`\
-   Великі файли\
-   `node_modules`

### **22. Що робити якщо випадково закомітили пароль?**

-   Видалити з історії\
-   Змінити пароль або токен

### **23. HTTPS vs SSH для GitHub --- що краще?**

SSH --- безпечніше та зручніше для частого використання.\
HTTPS --- універсальний варіант.

### **24. Що таке Personal Access Token?**

Токен для авторизації GitHub через HTTPS або API.

## 7. GitHub

### **25. Що таке Pull Request?**

Запит на злиття змін із можливістю перегляду коду (code review).

### **26. Яка різниця між Issues та Pull Requests?**

-   **Issues** --- задачі та проблеми.\
-   **Pull Requests** --- пропозиції змін у коді.

### **27. Що таке Fork і навіщо він потрібен?**

Копія чужого репозиторію у своєму акаунті.\
Використовується для внеску у сторонні проєкти.

### **28. Що таке GitHub Actions?**

Система автоматизації: тести, деплой, CI/CD.

## 8. Best Practices

### **29. Як часто треба робити коміти?**

Часто, але логічними порціями --- один коміт = одна зміна.

### **30. Скільки часу має жити feature branch?**

Коротко: кілька днів або до завершення задачі. Чим довше --- тим більше
конфліктів.

### **31. Як організувати git workflow в команді?**

-   Використовувати гілки\
-   Робити code review\
-   Часті pull\
-   CI/CD\
-   Git-flow або Trunk-based development

### **32. Чи треба комітити node_modules або інші залежності?**

**Ні.**\
Потрібно зберігати лише файли залежностей (package.json,
requirements.txt тощо).
